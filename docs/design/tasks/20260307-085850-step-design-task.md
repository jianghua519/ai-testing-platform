# step级执行控制与回传设计说明

## 背景

到上一轮为止，worker 已经具备：

- job 级结果回传
- HTTP publisher
- compile -> execute -> publish 主链路

但还缺两个关键能力：

- 不能在 step 边界做动态控制
- 不能把 step 结果逐条实时回传

这会直接限制你要的场景：前一个 step 跑完后，人或 AI 想实时改下一个 step，再继续执行。

## 设计目标

- 在执行引擎中建立 step 执行前控制窗口。
- 在 worker 中建立 step 完成即回传的机制。
- 支持 in-memory 级别的 `replaceNextStep`、`skipNextStep`、`pause/resume`。
- 用一次真实运行流证明“step1 完成后热替换 step2 再执行”已经成立。

## 一、adapter 层改动

### 1.1 新增 StepExecutionController 与 StepLifecycleObserver

[types.ts](/home/jianghua519/ai-web-testing-platform-v2/packages/playwright-adapter/src/types.ts) 新增：

- `StepControlDecision`
- `StepExecutionController`
- `StepLifecycleObserver`
- `ExecutionSession.controller`
- `ExecutionSession.observer`

这让执行引擎在真正调用 executor 之前，有机会做以下动作：

- 正常执行
- 跳过当前 step
- 替换当前即将执行的 step

同时在 step 完成后，也能把结果回调给 worker 层。

### 1.2 ExecutionEngine 改为 step 前控制 + step 后观察

[execution-engine.ts](/home/jianghua519/ai-web-testing-platform-v2/packages/playwright-adapter/src/runtime/execution-engine.ts) 本轮关键变化：

- `beforeStep()` 在 executor 之前执行
- 如果 controller 返回 `replacementStep`，则执行替换后的 step
- 如果 controller 返回 `skip`，则直接构造 skipped result
- `onStepStarted` / `onStepCompleted` 已接入 step 生命周期
- `buildSkippedResults()` 也会同步触发 step 完成观察，避免分支跳过时丢失 step 结果

这使得 step 控制已经从“概念设计”变成“运行时真实入口”。

### 1.3 InMemoryStepController

[step-controller.ts](/home/jianghua519/ai-web-testing-platform-v2/packages/playwright-adapter/src/runtime/step-controller.ts) 提供了最小控制实现：

- `pause()`
- `resume()`
- `replaceNextStep(stepId, replacementStep)`
- `skipNextStep(stepId, reason)`

这个实现不是最终的控制面协议，但它证明运行时控制模型成立。

## 二、worker 层改动

### 2.1 step 级回传协议

[reporting/types.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/reporting/types.ts) 新增：

- `StepResultPayload`
- `StepResultReportedEnvelope`
- `ResultPublisher.publishStep()`

这意味着 worker 不再只能发 job 级结果，而是可以逐条发 step 级结果。

### 2.2 EnvelopeFactory 扩展

[result-envelope.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/reporting/result-envelope.ts) 当前既支持：

- `buildJobResult()`
- `buildStepResult()`

step payload 当前包含：

- `compiled_step_id`
- `source_step_id`
- `status`
- `started_at`
- `finished_at`
- `duration_ms`
- `error`
- `artifacts`
- `extracted_variables`

### 2.3 HttpResultPublisher 扩展

[http-publisher.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/reporting/http-publisher.ts) 现在支持两类发布：

- `publish()` -> `job.result_reported`
- `publishStep()` -> `step.result_reported`

所以当前 HTTP publisher 已经从“job 级一次性回传”升级为“step + job 双层回传”。

### 2.4 PublishingStepObserver

[step-result-observer.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/reporting/step-result-observer.ts) 负责把 adapter 的 step 完成回调转成 worker 的 step 级发布动作。

`WebJobRunner` 已接入这个 observer，见 [web-job-runner.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/job-runner/web-job-runner.ts) 。

## 三、worker 装配改动

### 3.1 SessionManager

[session-manager.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/session/session-manager.ts) 已支持把 `controller` 和 `observer` 注入 `ExecutionSession`。

### 3.2 createWebWorker

[create-worker.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/bootstrap/create-worker.ts) 现在可以接收可选 `controller`，默认会构造一个 `InMemoryStepController`。

这让“step 控制能力”正式进入 worker 的默认装配链路，而不是孤立工具类。

## 四、契约更新

本轮扩展了 [asyncapi.yaml](/home/jianghua519/ai-web-testing-platform-v2/contracts/asyncapi.yaml) ：

- 新增 channel：`runner.step_results.reported`
- 新增 schema：`StepResultPayload`
- 新增 message：`StepResultReported`

这让 step 级回传不再只是代码约定，而是契约的一部分。

## 五、真实运行验证

本轮做了一次真实运行流验证，关键过程是：

1. 用 fixture 生成两步 plan：`open-home`、`open-dashboard`
2. 先 compile，得到 step2 的编译结果模板
3. 启动 `WebJobRunner.run()`
4. 当收到 step1 的 `step.result_reported` 时，调用 `controller.replaceNextStep('open-dashboard', replacement)`
5. step2 实际执行的是热替换后的 URL
6. 本地 HTTP server 收到两条 `step.result_reported` 和一条 `job.result_reported`

关键结果：

- `visitedUrls[1] === https://example.com/dashboard-patched`
- event types 顺序为：
  - `step.result_reported`
  - `step.result_reported`
  - `job.result_reported`
- `open-home` 和 `open-dashboard` 都为 `passed`

这证明：

- step 级回传已经成立
- 在 step1 完成后替换 step2 已经成立
- 替换后的 step2 确实执行了

## 风险

- 当前控制能力还是 in-memory，本轮没有实现远程双向控制协议。
- 当前只支持“在 step 执行前替换/跳过”，不支持对正在执行中的 step 做中途改写。
- 当前运行验证不是容器内验证，也不是真实浏览器 E2E。

## 验证计划

- 运行 `npm run typecheck`。
- 运行一次真实流，验证 step1 后热替换 step2 并继续执行。
- 检查本地 HTTP 接收端是否收到两条 step 事件和一条 job 事件。
- 运行 `bash ./scripts/validate_docs.sh` 和 `bash ./scripts/validate_contracts.sh`。
- 检查容器与真实浏览器运行入口缺失情况。
