# 远程step控制协议设计说明

## 背景

上一轮已经把 step 级执行控制与 step 级结果回传落到运行时，但控制动作仍然只能靠 `InMemoryStepController` 在进程内完成。这能证明能力模型成立，却还不能回答一个更关键的问题：

- 控制面如何远程拦截下一个 step
- AI 或人工如何在前一个 step 结束后修改下一个 step 的定义
- worker 如何在不重启进程的情况下应用这个新定义

因此本轮目标不是再扩展 step 结果模型，而是把 step 控制入口推进成真正的远程协议。

## 设计目标

- 保持现有 `StepExecutionController` 边界不变，不重写 adapter 执行引擎
- 用最小协议把远程控制接入 `web-worker`
- 让控制策略按 job metadata 构造，避免不同 job 共享同一个运行时控制对象
- 通过一次真实运行流证明“远程替换 step2 后继续执行”成立

## 一、控制协议模型

### 1.1 请求模型

[control/types.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/control/types.ts) 定义了 `StepControlRequest`。worker 在 step 执行前会把这些信息发给控制端：

- `job_id` / `run_id` / `run_item_id` / `attempt_no`
- `tenant_id` / `project_id`
- `trace_id` / `correlation_id`
- `compiled_step_id` / `source_step_id` / `step_name`
- `page_url`
- `compiled_step`

把完整 `compiled_step` 发过去的原因很直接：控制面或 AI 要修改“下一个 step”，就必须先拿到当前即将执行的 step 结构，而不是只拿一个 step id。

### 1.2 决策模型

当前 `StepControlResponse` 支持四种动作：

- `execute`：按原 step 执行
- `pause`：等待一段时间后重新请求控制决策
- `skip`：跳过这个 step
- `replace`：用 `replacement_step` 替换当前即将执行的 step

这套动作刻意保持在 step 边界内，不修改正在执行中的 step。原因是当前 `ExecutionEngine` 的安全控制窗口就在 `beforeStep()`，把修改范围限定在“尚未执行的 step”更稳，也更容易审计。

## 二、HttpStepController

### 2.1 控制器实现

[http-step-controller.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/control/http-step-controller.ts) 实现了 `HttpStepController`，核心逻辑是：

1. 在 `beforeStep()` 中构造 `StepControlRequest`
2. 调控制端 HTTP 接口
3. 解析返回的 `StepControlResponse`
4. 根据响应执行：
   - 直接执行
   - 休眠后重试
   - 跳过
   - 替换 step

这里没有引入 WebSocket 或消息总线，原因是这轮目标是先让 worker 与远程控制面形成最小可运行协议。按 step 边界做请求/响应，已经足够支持“前一步结束后，实时修改下一步”。

### 2.2 URL 模板

控制端 endpoint 支持 URL 模板。例如：

- `/api/v1/agent/jobs/{job_id}/steps/{source_step_id}:decide`

控制器会在发请求前替换这些占位符：

- `{job_id}`
- `{run_id}`
- `{run_item_id}`
- `{source_step_id}`
- `{compiled_step_id}`

这样代码和内部控制契约保持一致，而不是把 job/step 标识再额外塞进 query string。

### 2.3 fail-open 策略

当前 `HttpStepController` 提供 `failOpen` 开关：

- `false`：控制端异常时直接抛错，阻止 step 执行
- `true`：控制端异常时按原 step 执行

默认通过环境变量装配时为 `false`，因为只要显式启用了远程控制模式，通常就意味着这条链路是强依赖，而不是可有可无的增强项。

## 三、worker 装配改动

### 3.1 StepControllerFactory

[control/types.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/control/types.ts) 新增了 `StepControllerFactory` 与 `StepControllerProvider`。

这样 `WebJobRunner` 不再只接受一个预先构造好的 controller，而是支持：

- 直接传一个静态 controller
- 传一个 factory，由 worker 在每个 job 开始时按 metadata 创建 controller

### 3.2 WebJobRunner

[web-job-runner.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/job-runner/web-job-runner.ts) 本轮的关键变化：

- 在 `run(job)` 开头先生成 `metadata`
- 再根据 `controllerProvider` 解析当前 job 的 controller
- 把这个 controller 注入 `ExecutionSession`

这让远程控制能力和每个 job 的身份绑定在一起，避免不同 job 共享一个错误的控制上下文。

### 3.3 createWebWorker

[create-worker.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/bootstrap/create-worker.ts) 现在支持：

- `controller`
- `controllerFactory`
- 未显式传入时，通过 [create-controller.ts](/home/jianghua519/ai-web-testing-platform-v2/apps/web-worker/src/control/create-controller.ts) 按环境变量装配

支持的环境变量：

- `WEB_WORKER_STEP_CONTROL_MODE=inmemory|http`
- `WEB_WORKER_STEP_CONTROL_ENDPOINT`
- `WEB_WORKER_STEP_CONTROL_TIMEOUT_MS`
- `WEB_WORKER_STEP_CONTROL_AUTH_TOKEN`
- `WEB_WORKER_STEP_CONTROL_PAUSE_POLL_INTERVAL_MS`
- `WEB_WORKER_STEP_CONTROL_FAIL_OPEN`

## 四、控制面内部契约

本轮新增了内部控制决策接口契约 [openapi.yaml](/home/jianghua519/ai-web-testing-platform-v2/contracts/openapi.yaml) ：

- `POST /api/v1/agent/jobs/{job_id}/steps/{source_step_id}:decide`

请求体是 `WorkerStepControlRequest`，响应体是 `WorkerStepControlDecision`。

这个接口不是终端用户 API，而是控制面与 worker 的内部运行时接口，用来在每个 step 开始前给出执行决策。

## 五、真实运行验证

本轮做了一次真实运行流验证，结构如下：

- 本地 HTTP server A：接收 `step.result_reported` 和 `job.result_reported`
- 本地 HTTP server B：作为远程 step 控制端
- `WebJobRunner` 使用 `HttpResultPublisher + HttpStepController`
- 浏览器执行层仍使用 fake browser 对象

验证过程：

1. worker 执行 `open-home`
2. `open-home` 的 step 结果被回传到结果接收端
3. 结果接收端把 `allowPatch` 标记置为 `true`
4. worker 在执行 `open-dashboard` 之前，请求控制端决策
5. 控制端先返回一次 `pause`
6. worker 重试请求后，控制端返回 `replace`，把 `inputResolved.value` 改成 `https://example.com/dashboard-remote-patched`
7. worker 执行热更新后的 step2，并继续回传 step 结果与最终 job 结果

这证明当前协议已经能做到：

- 远程在 step 边界拦截执行
- 远程返回修改后的 next step
- worker 应用修改并继续执行

## 风险

- 当前协议是 step 边界请求/响应，不是持久双向流式通道
- 当前 `replace` 的输入是 `compiled_step`，还不是 source DSL 重编译协议
- 当前验证没有真实浏览器，也没有真实控制面服务

## 验证计划

- 运行 `npm run typecheck`
- 启动本地结果接收端和控制端，执行一次真实运行流
- 验证控制端收到 step 决策请求，并成功返回 `pause` + `replace`
- 验证 step2 实际访问 URL 已变成远程修改后的值
- 运行 `bash ./scripts/validate_docs.sh` 和 `bash ./scripts/validate_contracts.sh`
- 检查容器入口缺失情况
