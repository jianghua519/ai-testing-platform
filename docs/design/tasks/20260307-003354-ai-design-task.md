# AI自动化测试工具主要技术选型说明

## 背景

上一轮已经确定了平台总体架构：服务器控制面 + 异构执行面 + AI-native + 记忆内建 + Web/API 双测试域。本轮要把“架构语言”收敛成“可实施的技术栈”。

技术选型的目标不是寻找某个“最先进”的技术，而是给后续 12-18 个月的实现提供一个稳定基线：

- Phase 1 能尽快落地核心闭环。
- 技术栈数量受控，认知成本可管理。
- 能支撑多租户、跨平台 agent、step 级 Web 控制和 AI 能力接入。
- 后续可以扩展，不因早期选型锁死架构。

## 选型原则

- 主路径优先：先保证 Web/API 自动化主路径可靠，再考虑高级能力。
- 运行时可控：执行层优先选择稳定、可调试、可审计的技术。
- 少而强：Phase 1 只引入必要的 3 种主要运行时语言。
- 渐进增强：复杂基础设施如独立向量库、复杂工作流引擎、分析数据库不在第一阶段强上。
- 开源优先：核心基础设施优先选成熟开源组件，降低部署和私有化门槛。

## 结论总览

| 模块 | 推荐选型 | 说明 |
| --- | --- | --- |
| Web Console | Next.js + React + TypeScript | 管理台、测试设计台、报告查看台统一前端技术栈 |
| 控制面核心服务 | Go | 高并发、跨平台、部署简单、适合调度与 agent 管理 |
| AI 编排服务 | Python + FastAPI | AI 工具链成熟，便于模型编排与实验迭代 |
| Agent Core | Go | 单二进制跨平台发布，适合 Windows / Linux / macOS / 容器 |
| Web 执行引擎 | Node.js + Playwright | 浏览器自动化生态最成熟，适合 step 级控制 |
| API 执行引擎 | Go 原生 HTTP/gRPC Worker | 更轻量，适合高并发接口测试和链路编排 |
| 主数据库 | PostgreSQL | 交易数据、配置数据、结构化记忆统一落库 |
| 缓存 / 分布式锁 | Redis | 队列旁路缓存、幂等辅助、锁和限流 |
| 消息队列 | RabbitMQ | 与现有 AsyncAPI 方向一致，适合任务派发和事件流 |
| 对象存储 | S3 API / MinIO | 截图、视频、trace、HAR、报告导出 |
| 向量 / 记忆检索 | PostgreSQL + pgvector | Phase 1 降低复杂度，后续可迁移独立向量库 |
| 可观测性 | OpenTelemetry + Prometheus + Loki + Tempo + Grafana | 覆盖 metrics / logs / traces |
| 容器编排 | Docker Compose + Kubernetes | 开发单机，生产集群 |
| 插件扩展 | gRPC/HTTP 插件契约 + OCI 分发 | 先做远程插件，不急着上 WASM |

## 为什么是三种主要语言

### Go

用于：

- 控制面核心服务
- Agent Core
- API 执行 worker
- 调度、状态机、队列消费者、审计、权限边界

选择原因：

- 单二进制分发对私有化部署、跨平台 agent 非常友好。
- 并发模型和内存占用适合调度系统、消息消费者和执行控制。
- 在多租户后台服务中，稳定性和资源效率优于 Node/Python。

不选择 Java / .NET 作为主栈的原因：

- 平台早期更需要轻量部署和跨平台 agent 分发，Go 更直接。
- 团队如果规模还不大，引入重量级企业栈会提高交付成本。

### TypeScript

用于：

- Web Console
- Web 执行引擎的 step DSL 与 Playwright worker
- 前端可视化用例设计器

选择原因：

- Playwright 生态最成熟，TypeScript 是一等公民。
- 前端和 Web 执行 DSL 共用类型系统，能减少 step 定义漂移。
- 浏览器场景下，TypeScript 的开发体验和生态明显优于 Go/Python。

### Python

用于：

- AI 编排服务
- 模型网关集成
- prompt / tool / retrieval 实验层

选择原因：

- AI 生态最成熟，模型适配、RAG、embedding、rerank、agent workflow 工具链丰富。
- 更适合快速试错，避免把 AI 逻辑硬编码进控制面主服务。

不把整个平台都放在 Python 的原因：

- 调度、agent、队列消费、跨平台分发这些部分更适合 Go。
- Web 执行与浏览器自动化仍然更适合 Playwright / TypeScript。

## 控制面技术选型

### 推荐

- 语言：Go 1.24+
- Web 框架：Gin
- 数据访问：`pgx` + `sqlc`
- 配置与服务启动：`cobra` + `viper` 或等价轻量方案
- 后台任务：基于 RabbitMQ 的 worker，不引入 Temporal
- API 契约：OpenAPI 驱动外部 REST API，内部服务之间优先 HTTP/gRPC 风格接口

### 理由

- Gin 足够轻，学习成本低，生态成熟。
- `sqlc` 能保证 SQL 显式可控，适合多租户边界和复杂查询，不把关键查询隐藏在 ORM 黑盒里。
- 当前平台的执行状态机、调度、重试虽复杂，但还没到必须引入 Temporal 的程度；先用 RabbitMQ + 显式状态机更可控。

### 当前不选

- 不选大型微服务框架：早期会让边界看上去“标准”，但会加重框架负担。
- 不选 ORM 主导模式：多租户和审计约束下，SQL 显式更稳。
- 不选先上 gRPC-only：浏览器和第三方集成仍然更适合 REST 优先。

## Web Console 技术选型

### 推荐

- Next.js 15 + React 19 + TypeScript
- 组件层：shadcn/ui + Tailwind CSS
- 数据获取：TanStack Query
- 表单：React Hook Form + Zod
- 可视化编排：React Flow
- 文本 / 富编辑：TipTap

### 理由

- Console 不是纯 CRUD 页面，还要承载需求分析、step 编辑、执行观察、报告查看和 AI 交互面板。
- Next.js 适合 BFF 风格和复杂管理台路由。
- React Flow 非常适合可视化展示 Web step 流程和 API 场景链路。

### 当前不选

- 不选纯 Vue 栈：不是不能做，而是与 Playwright / TypeScript worker 的类型复用收益更低。
- 不选桌面优先前端：当前产品核心是服务器部署和浏览器访问。

## Agent 技术选型

### 推荐

- Agent Core：Go
- 配置格式：YAML + 环境变量覆盖
- 通信协议：HTTPS + WebSocket 事件通道，任务派发仍以控制面队列模型为准
- 打包方式：
  - Windows：`.exe` + service wrapper
  - Linux：systemd service + tarball / deb / rpm
  - macOS：launchd + pkg
  - 容器：OCI image

### 理由

- Go 非常适合做跨平台守护进程和自升级 agent。
- Agent Core 不需要承担浏览器生态复杂性，它负责注册、拉取、调度、资源控制和产物上传。

## Web 测试执行技术选型

### 推荐

- Node.js 22 LTS
- Playwright
- TypeScript step DSL
- 浏览器制品管理：Playwright browsers + 可选企业镜像缓存

### 理由

- Playwright 对 Chromium / Firefox / WebKit 支持完整，天然适配 Windows / Linux / macOS。
- 它对 trace、video、network、locator、frame、storage state 的支持足够强，适合 step 级控制。
- Step DSL 用 TypeScript 定义，能让 Console 和执行器共享结构定义。

### Step 级控制实现建议

- Console 编辑的是结构化 step，不是自由脚本。
- 执行器把 step 转为 Playwright action 序列。
- 每个 step 记录：
  - locator
  - 入参
  - 断言
  - timeout
  - retry
  - artifact policy
- 失败后返回 step 级错误对象，而不是只返回整段脚本失败。

### 当前不选

- 不选 Selenium 作为主引擎：在 step 级可观测性、现代浏览器能力和开发体验上不如 Playwright。
- 不选 Cypress：跨浏览器和执行形态限制更大，不适合作为平台级执行底座。

## API 测试执行技术选型

### 推荐

- API Worker：Go
- HTTP client：原生 `net/http` + `resty` 辅助封装
- gRPC：`grpc-go`
- Schema 校验：JSON Schema / OpenAPI 驱动校验
- 场景编排：内部 DSL + 变量抽取 + 前后置依赖

### 理由

- API 执行不依赖浏览器，没必要强绑定 Node。
- Go 适合高并发、轻量接口执行和链路编排。
- 未来可以很自然扩展到 gRPC、WebSocket、GraphQL。

## AI 编排技术选型

### 推荐

- Python 3.12+
- FastAPI
- Pydantic
- LiteLLM 作为模型适配层
- LangGraph 作为多步骤 AI workflow 编排层

### 理由

- FastAPI + Pydantic 在结构化输入输出、工具接口暴露上足够成熟。
- LiteLLM 能屏蔽多模型供应商差异，方便切换 OpenAI、Azure OpenAI、私有模型网关。
- LangGraph 适合表达“需求解析 -> 检索记忆 -> 生成候选用例 -> 人审”的多阶段流程。

### 当前不选

- 不把 AI 逻辑直接写进 Go 控制面：会让主服务迭代变慢。
- 不急着引入重量级 AI 中台：先把 AI 编排层做成独立服务即可。

## 记忆系统技术选型

### Phase 1 推荐

- PostgreSQL 16
- `pgvector`
- 结构化事实表 + 事件表 + 向量表组合

### 理由

- 记忆最初规模不会大到必须上专用向量库。
- 使用 Postgres 可以把元数据过滤、租户隔离、事务一致性和向量检索放在同一套系统里。
- 对需求、case、failure、report 的统一检索会更简单。

### Phase 2 预留

- 如果向量规模、检索延迟、复杂检索需求上来，再拆到 Qdrant 或 Milvus。

## 数据与基础设施选型

### 主数据库

- PostgreSQL 16

原因：

- 多租户业务数据、配置数据、审计、调度状态都适合放在 Postgres。
- 可扩展到 `pgvector`、RLS、JSONB、全文检索。

### 消息队列

- RabbitMQ

原因：

- 与当前仓库的 AsyncAPI 方向一致。
- 任务派发、结果上报、事件通知的 at-least-once 语义适配度高。
- 运维复杂度低于 Kafka。

### 缓存与锁

- Redis 7

原因：

- 做热点缓存、幂等缓存、限流计数、短时分布式锁足够合适。

### 对象存储

- S3 API 兼容对象存储
- 开发环境使用 MinIO

原因：

- 截图、视频、trace、HAR、导出报告都天然适合对象存储。

## 可观测性选型

### 推荐

- OpenTelemetry SDK
- Prometheus
- Loki
- Tempo
- Grafana

### 理由

- metrics / logs / traces 能形成完整关联。
- step 级执行和 AI 调用都需要 trace 化，否则排障成本会很高。

## 身份认证与权限

### 推荐

- 外部 SSO 协议：OIDC / SAML
- 自托管参考实现：Keycloak
- 细粒度权限：平台内部 RBAC + 项目级 scope

### 理由

- 企业客户对 SSO 和权限隔离要求高。
- Keycloak 不是最轻的，但作为参考实现更稳，更符合私有化部署预期。

## 插件与扩展机制选型

### 推荐

- 插件交互协议：gRPC 或 HTTP
- 插件分发：OCI 镜像或制品仓库
- 插件元数据：JSON Schema / YAML manifest

### 原因

- 远程插件边界比进程内插件更安全，隔离更清晰。
- 对于断言器、报告器、企业连接器，独立服务形态更容易治理权限和资源。

### 当前不选

- 不在 Phase 1 上 WASM 插件系统：隔离和跨语言很诱人，但生态、调试、团队理解成本还偏高。

## 开发与交付基础设施

### 推荐

- 开发环境：Docker Compose
- 生产环境：Kubernetes
- CI：GitHub Actions / GitLab CI 均可，核心是容器镜像构建 + 文档 / 契约校验 + 测试执行
- 镜像仓库：GHCR / Harbor

## 分阶段落地建议

### Phase 1

- Go 控制面核心服务
- Go Agent Core
- Node.js + Playwright Web 执行
- Go API Worker
- Python AI 编排服务
- PostgreSQL + RabbitMQ + Redis + MinIO
- OpenTelemetry + Prometheus + Loki + Tempo

### Phase 2

- 引入独立向量库（如果需要）
- 扩展 macOS / Windows 专项执行能力
- 插件生态化
- 更复杂的 AI 工作流与审批策略

## 最终建议

如果目标是“尽快把平台跑起来，同时保留长期扩展性”，建议把核心技术栈收敛到：

- `Go`：控制面、Agent Core、API Worker
- `TypeScript + Playwright`：Web 执行与前端 DSL 共享
- `Python + FastAPI + LiteLLM + LangGraph`：AI 编排层
- `PostgreSQL + pgvector + RabbitMQ + Redis + MinIO`：Phase 1 基础设施底座
- `OpenTelemetry + Prometheus + Loki + Tempo + Grafana`：统一观测

这套组合不是最“少语言”的方案，但在你的需求下是最均衡的方案：执行层可控、agent 可跨平台、AI 能快速迭代、Web step 级控制可落地、API 测试不会被浏览器技术栈绑死。
