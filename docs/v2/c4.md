# V2 Architecture (C4)

This document is normative. Any V2 implementation must remain consistent with it.

## Goals

- Clear boundary between Control Plane and Data Plane.
- Multi-tenant isolation by default.
- Contract-first delivery (OpenAPI + AsyncAPI) with versioning.
- Operational readiness (observability, auditability, disaster recovery) as release gates.

## C4: Context

Actors:

- QA Engineer / Test Engineer: create assets, run executions, view reports.
- Admin: manage tenants/projects/users/roles, quotas, integrations.
- CI System: triggers executions and fetches results via API.

External systems:

- Identity Provider (IdP): OIDC/SAML for SSO.
- Issue Tracker: Jira/Zentao/etc for defect sync (optional).
- Object Storage (S3-compatible): evidence storage (MinIO in dev; S3 in prod).

```mermaid
  user[User/Admin/CI] --> gw[API Gateway]

  dp --> obs
```

## C4: Containers

### Control Plane (authoritative for business state)

- API Gateway: routing, auth enforcement entrypoint, rate limiting.
- IAM Service: authn/authz, RBAC, tokens, SSO integration.
- Orchestrator API: Run/TestPlan lifecycle, state machines, idempotency, quota enforcement.
- Scheduler Worker: scheduled runs, retries, compensation.
- Reporting Service: report job orchestration + renderers (artifact generation).
- Billing/Metering Service: usage aggregation (token/minutes/storage) and quota evaluation.
- Audit Service: append-only audit events and query APIs (can be a module initially).

### Data Plane (execution only)

- Runner Pool: executes jobs (Playwright), produces artifacts, reports results/events.
- Local Agent (optional): executes in customer network; connects via secure channel.

### Shared infrastructure

- Postgres: source of truth for control plane state.
- Message Queue: job dispatch + event streaming.
- Object Storage: evidence (screenshots, traces, videos, exports).
- Redis (optional): caching, distributed locks, rate limit counters.
- Observability: metrics/logs/traces storage (e.g. Prometheus + Loki + Tempo/Jaeger).

## C4: Deployment

Deployment modes:

- Dev: single-node Docker Compose.
- Prod: Kubernetes (or VM-based) with:
  - stateless control plane pods (horizontally scalable)
  - runner pools sized independently (autoscaling by queue depth)
  - managed Postgres + managed object storage preferred

Key requirements:

- All inbound traffic terminates TLS at gateway/ingress.
- Secrets are managed via secret manager; never stored in repo.
- Audit events are append-only and retained per policy.
- Evidence storage uses per-tenant key prefix and signed URLs for download.

## Boundary Rules (non-negotiable)

- Control Plane owns: auth, tenant boundaries, quotas, state machines, persistence.
- Data Plane owns: execution, artifact capture, best-effort local retries.
- Runner must not query business tables directly.
- All external contracts are versioned and backward compatible within a major version.
