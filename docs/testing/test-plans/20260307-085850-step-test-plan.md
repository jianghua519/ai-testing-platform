# step级执行控制与回传测试计划

## 测试范围

- `playwright-adapter` 的 controller / observer / execution engine 改动。
- `web-worker` 的 step 级 envelope 与 publisher 扩展。
- `AsyncAPI` 中新增的 step 级事件契约。
- 一次真实运行流：step1 完成后替换 step2，再执行 step2。
- 文档校验和契约校验。

## 覆盖风险

- step 控制钩子存在但没有真正生效。
- step 结果没有逐条实时回传。
- 替换下一个 step 只是静态配置，不是运行时动态生效。
- step 事件契约与代码不一致。

## 测试用例

1. 检查 `packages/playwright-adapter/src/types.ts` 是否存在 `StepExecutionController`、`StepLifecycleObserver`。
2. 检查 `InMemoryStepController` 是否存在并支持 `replaceNextStep` / `skipNextStep` / `pause` / `resume`。
3. 检查 `ResultPublisher` 是否新增 `publishStep()`。
4. 执行 `npm run typecheck`。
5. 执行一次本地真实运行流，验证 step1 后热替换 step2。
6. 检查本地 HTTP server 收到的事件类型是否为两条 `step.result_reported` 和一条 `job.result_reported`。
7. 执行 `bash ./scripts/validate_docs.sh`。
8. 执行 `bash ./scripts/validate_contracts.sh`。

## 通过标准

- step 控制与 step 回传已在代码中打通。
- 运行流中 step2 被成功热替换并执行。
- 事件顺序与数量符合预期。
- 类型检查、文档校验、契约校验通过。
